# 灾难恢复手册

## 目标与权威数据源

- 目标恢复点（RPO）：已推送到远端 Git 仓库的最后一个提交。
- 目标恢复时间（RTO）：静态阅读站 2 小时内恢复；在线批阅链路 1 个工作日内恢复。
- 代码、Markdown、动画、声明式服务配置和 n8n 工作流以私有或受控 GitHub 仓库为权威来源。
- GitHub Pages 构建产物、服务器工作目录、缓存和搜索索引均可丢弃，不能作为唯一副本。
- 批阅清单位于用户浏览器会话中，不承诺跨设备或灾难恢复。

## 前置条件与秘密清单

干净环境需要 Git、Node.js 22 和 npm。以下秘密只允许从批准的秘密管理器恢复或按服务商流程重新签发，不得写入 Git：

- GitHub OAuth 客户端密钥；
- 登录网关与 n8n 之间的内部令牌；
- n8n 与隔离执行器之间的内部令牌；
- 仓库专用 Deploy Key；
- Codex 登录状态或重新认证凭据；
- DNS 服务商的最小权限令牌。

恢复前确认各凭据仍限定到单一仓库、必要域名或必要接口。凭据不可恢复时应撤销旧值并重新签发。

## 静态站恢复

1. 在干净机器克隆权威仓库并检出目标提交。
2. 运行 `npm ci`。
3. 运行 `npm run check`，确认测试与 VitePress 构建通过。
4. 在 GitHub Pages 中选择 GitHub Actions 作为发布源。
5. 推送目标提交或重新运行对应 Pages 工作流。
6. 核对工作流展示的提交 SHA 与目标提交一致。

## 在线批阅恢复

1. 从仓库恢复 `automation/` 下的声明式服务文件与 n8n 工作流。
2. 在新的部署服务器创建无登录、低权限的独立服务账号。
3. 从秘密管理器注入秘密；不要复制旧服务器的整个主目录。
4. 导入并发布版本化 n8n 工作流，部署登录网关、执行器和发布器。
5. 恢复仅允许必要来源、端口和仓库分支的网络与 Git 权限。
6. 先运行健康检查和无副作用的非法负载测试，再提交一条受控批阅烟雾任务。
7. 确认任务只生成允许路径内的差异，并经过 PR 门禁后合并。

## DNS 与 TLS

1. 从 DNS 服务商恢复指向 GitHub Pages 的声明式记录；实际域名通过部署环境配置提供。
2. 在 GitHub Pages 中重新登记自定义域名。
3. 等待证书签发后启用强制 HTTPS。
4. 验证根页面、嵌套文章、静态演示和回调入口均使用 HTTPS。

## 完整性验证

- `npm ci` 和 `npm run check` 在干净工作树中通过；
- 生产首页、至少一篇文章和一个独立 HTML 演示返回成功；
- 搜索、专注阅读开关和移动端导航可用；
- 登录失败路径不泄露内部错误；
- 受控批阅任务的提交 SHA、PR 检查和 Pages 部署 SHA 可追踪；
- 远端仓库仍是唯一权威源，部署服务器没有未同步的源码或工作流修改。

每季度至少执行一次静态站干净构建演练；在线批阅链路至少每半年执行一次受控恢复演练并把证据记录到项目跟踪系统。未实际演练的部分必须明确标记为未验证。

## 回滚

优先使用 GitHub 的历史提交创建回滚 PR，不直接改写默认分支历史。合并回滚后等待 Pages 使用该提交重新部署，并重复完整性验证。服务端恢复失败时先关闭批阅入口，保留公开静态阅读站。
